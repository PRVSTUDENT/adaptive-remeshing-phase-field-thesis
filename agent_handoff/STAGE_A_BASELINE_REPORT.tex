\section{WP0 Environment and Toolchain}

The local environment inspection recorded Abaqus 2024, Abaqus Python 3.10.5, Windows 11, 12 logical processors, approximately \SI{23.35}{GiB} physical memory, GNU Fortran availability, and the absence of a usable Intel Fortran toolchain on a plain shell path. The successful user-subroutine smoke gate then established the required clean-shell sequence: load Visual Studio 2022 Build Tools with \texttt{vcvars64.bat}, then Intel oneAPI with \texttt{setvars.bat intel64}. In that environment, Abaqus used Intel \texttt{ifx} 2026.0.0 for compilation and Microsoft \texttt{LINK} 14.44.35226.0 for linking.

This is a local observed toolchain pass only. It is not an official Abaqus 2024--oneAPI 2026 support claim and it does not validate any phase-field formulation.

\section{WP1 One-Element Verification}

The unchanged one-element example passed the technical gate and the source-defined scientific check under provisional numerical tolerances. The validator opens the unchanged ODB read-only, checks all \texttt{SDV1}--\texttt{SDV16}, processes every frame and integration point, and writes CSV, JSON, and Markdown evidence.

The one-element verification established:
\begin{itemize}
  \item plane-strain undamaged elasticity;
  \item degraded stress using the phase field applied by the displacement element;
  \item homogeneous phase-field relation;
  \item monotonic history field;
  \item no phase-field healing during unloading;
  \item consistency across the four integration points.
\end{itemize}

The maximum phase-relation absolute error was approximately \(5.98\times10^{-7}\). The undamaged-stress absolute errors remained below \(1.90\times10^{-6}\). The largest \texttt{SDV14}-\texttt{SDV15} difference was approximately \(4.88\times10^{-3}\), interpreted as a staggered-update difference between the phase field used by the displacement solve and the phase-element value, not as automatic source corruption.

\section{Gate A3 Scientific-Comparison Status}

The first Gate A3 validator is now available at \texttt{scripts/validation/check\_molnar\_single\_notch.py}. It reads the extracted RF--U data, matched-state contour CSVs, derived reference files, tolerance configuration, and the ODB. Its current classification is:

\begin{center}

This is not a scientific mismatch. It means the simulation-side diagnostics are available, but the numeric RF--U coordinates corresponding to the appropriate Molnar Fig.~7 curve have not yet been digitized or otherwise supplied. The original element calculation does not include an explicit clipping operation, so a small discrete overshoot is possible. This does not invalidate the technical gate, but it is now a required scientific diagnostic:
\begin{itemize}
  \item preserve the raw value;
  \item determine its element, integration point, and first occurrence;
  \item report both raw and display-only clipped contours;
  \item check whether \texttt{SDV14}, \texttt{SDV15}, or both overshoot;
  \item verify whether overshoot is limited to the final unstable propagation stage;
  \item do not silently clip values in quantitative calculations.
\end{itemize}

The first \texttt{SDV15} overshoot occurs in Step-2, frame 13, at \(U_2=0.006300\), element 8196, integration point 2, with value 1.0057975. \texttt{SDV14} also overshoots at the same frame and integration point. The maximum absolute staggered difference \(|\mathrm{SDV14}-\mathrm{SDV15}|\) is 0.0374441 at Step-2, frame 15, \(U_2=0.006500\), element 8188, integration point 3.

The maximum \texttt{SDV15} overshoot occurs at the final frame, Step-2 frame 20, \(U_2=0.007000\), element 8196, integration point 2, with value 1.010493 and overshoot 0.010493. Overshoot affects 11 unique integration points for both \texttt{SDV14} and \texttt{SDV15}. For \texttt{SDV15}, the overshoot duration spans global frames 64--71, corresponding to \(U_2=0.006300\) through \(U_2=0.007000\); all 11 affected points remain above one in the final frame. This supports the current interpretation that the overshoot belongs to the final unstable propagation stage, not to the elastic or stable-damage response.

\texttt{SDV16} never decreases at persistent material points in the ODB scan. Raw \texttt{SDV15} has 583 small decreases above the declared numerical tolerance \(10^{-8}\), with worst drop \(2.27809\times10^{-4}\). These decreases must be interpreted in the context of staggered element updates and UMAT visualization storage before being labelled as genuine healing.

This location should be inspected before declaring a true irreversibility violation.

The external work calculated by trapezoidal integration of the extracted RF--U curve is:
\[
W_{\mathrm{ext}} = 2.5813743\times10^{-3}.
\]

The current ODB/extraction does not yet provide a validated global integration contract for degraded elastic energy from \texttt{SDV12}, undamaged elastic energy from \texttt{SDV13}, or the crack-surface functional. Those quantities require integration weights and element volume/area times thickness, and the crack-surface functional also requires a valid \(\nabla d\) reconstruction or controlled output-enabled run. \texttt{SDV16} remains the local history field \(H\), not global fracture energy.
